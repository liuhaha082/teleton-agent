Architecture

Teleton Agent is structured as a layered system: a Telegram bridge at the bottom, an agent runtime in the middle, and a plugin/tool registry on top. All components are wired together by TeletonApp at startup.

System Overview

The system is composed of the following layers, from bottom to top:

Telegram Network
The external Telegram infrastructure (MTProto or Bot API).
ITelegramBridge
Abstracts user mode (GramJS) and bot mode (Grammy) behind a single interface.
MessageHandler
Handles deduplication, rate limiting, and per-chat serialization queues.
AgentRuntime
Manages sessions, assembles the system prompt, calls the LLM, and drives the tool loop.
ToolRegistry
Applies scope filtering, Tool RAG selection, tool execution, and timeout enforcement. Backed by both built-in modules and hot-reloadable plugins or MCP servers.
MemorySystem
SQLite database with FTS5 full-text search and sqlite-vec cosine-similarity embeddings.
WebUI / API
Optional Hono HTTP server with a React dashboard.

TeletonApp (src/index.ts) is the root class. Its constructor initialises every subsystem synchronously, then start() connects the bridge and wires event handlers.

Telegram Bridge Layer

All Telegram I/O flows through the ITelegramBridge interface (src/telegram/bridge-interface.ts). Two concrete implementations exist behind the same interface:

ModeImplementationCapabilities
userGramJS MTProto clientFull Telegram API: dialogs, history, search, stories, profile editing, Stars, gifts, stickers, voice, NFT marketplace, folders
botGrammy Bot API clientSend/edit/delete messages, reactions, pins, inline keyboards, streaming responses via message drafts

The active mode is set in config.telegram.mode. The factory function createBridge() returns the correct implementation. The bridge can be swapped at runtime (e.g., to switch from user to bot) via MessageHandler.setBridge() without restarting the agent.

ITelegramBridge (simplified)
interface ITelegramBridge {
  connect(): Promise<void>;
  getMode(): "user" | "bot";

  // Identity
  getOwnUserId(): bigint | undefined;
  getMe(): Promise<BotInfo | undefined>;

  // Messaging
  sendMessage(options: SendMessageOptions): Promise<SentMessage>;
  editMessage(options: EditMessageOptions): Promise<SentMessage>;
  deleteMessage(chatId: string, messageId: number): Promise<boolean>;

  // Streaming (bot mode only)
  streamResponse?(chatId: string, textStream: AsyncIterable<string>): Promise<SentMessage>;

  // Events
  onNewMessage(handler: (msg: TelegramMessage) => void): void;
}

Message Handler

MessageHandler (src/telegram/handlers.ts) sits between the bridge and the agent runtime. It owns all pre-processing logic before a message reaches the LLM.

  • Deduplication: an in-memory Set of up to 500 recent chatId:messageId keys prevents double-processing duplicate MTProto events. In user mode, a persistent offset store (offset-store.ts) additionally tracks the highest processed message ID per chat.
  • Policy check (analyzeMessage): evaluates dm_policy, group_policy, allow_from, admin_ids, and require_mention from config to decide whether to respond.
  • Rate limiting: a RateLimiter enforces a global messages-per-second cap and a per-group messages-per-minute cap.
  • Per-chat queue (ChatQueue): messages for the same chat are processed serially via a promise chain. Up to 10 chats are processed concurrently; additional chats are back-pressured.
  • Feed storage: every message is stored to the SQLite feed (tg_messages, tg_chats, tg_users) regardless of whether the agent responds.
  • Plugin hooks: onMessage hooks from loaded plugins are fired fire-and-forget after feed storage.
  • Voice transcription: voice and audio messages are automatically transcribed before being sent to the agent.

Agent Runtime

AgentRuntime (src/agent/runtime.ts) drives the LLM interaction loop. Its processMessage() method handles a single conversation turn end-to-end.

Key responsibilities:

  • Session management: loads or creates a per-chat session from SQLite, applies daily reset policy, archives and summarises old sessions before reset.
  • System prompt assembly: calls buildSystemPrompt() from src/soul/loader.ts, composing soul, security rules, strategy, identity, user profile, memory context, and current user info into a single prompt string.
  • Context retrieval: the ContextBuilder runs a hybrid RAG search (FTS5 + vector) over past messages and knowledge chunks to inject relevant context into the user message envelope.
  • Tool RAG: when enabled, the ToolIndex uses embedding similarity to select the most relevant tools from the registry, rather than sending all tools to the LLM.
  • LLM call and tool loop: calls the provider (Anthropic, OpenAI, Gemini, etc.) via pi-ai, executing tool calls concurrently (up to TOOL_CONCURRENCY_LIMIT) and feeding results back until the model stops requesting tools.
  • Context compaction: a CompactionManager monitors context token usage. Near the context window limit it either truncates old messages or triggers an LLM-assisted summary to preserve the session.
  • Streaming: in bot mode, the agent can stream the LLM response token-by-token via Telegram message draft updates.

Tool Registry

ToolRegistry (src/agent/tools/registry.ts) is the central catalog of all callable tools. It is mode-aware, scope-aware, and database-backed.

Registration

Built-in tools are registered at startup via registerAllTools(). Each tool specifies a ToolScope and optional mode restriction ("user" or "bot"). Plugin tools are registered via registerPluginTools() and tracked by plugin name for hot-reload.

Scope Filtering

getForContext() returns the subset of tools visible to the LLM for a given invocation, based on:

ScopeVisible when
always / openAll contexts
dm-onlyDirect messages only
group-onlyGroup chats only
admin-onlySender is in admin_ids
allowlistSender is in allow_from or is admin
disabledNever (hidden from LLM and blocked at execution)

Scopes can be overridden per-tool at runtime via the tool_config SQLite table, without restarting. Group chats additionally respect ModulePermissions, which allows disabling or admin-gating entire tool modules per group.

Toolset Profiles

The active toolset profile (minimal, standard, trading, or full) filters the tool list by tag before passing it to the LLM. Tags such as core, workspace, finance, and social are assigned at registration time.

Execution

Tool execution in registry.execute() enforces scope and mode checks a second time (defense-in-depth), validates parameters against the tool's JSON Schema via pi-ai, then runs the executor inside a Promise.race() against a configurable timeout. All errors are caught and returned as { success: false, error }.

Plugin System

The PluginOrchestrator (src/plugin-orchestrator.ts) loads all external capability layers after the built-in modules:

  1. Built-in modules are loaded first via loadModules(). These include telegram, ton, dex, dns, exec, memory, workspace, and others.
  2. External plugins are discovered and loaded by loadEnhancedPlugins(). Each plugin is a separate npm package exporting a manifest, tools(sdk), and optionally migrate(db), configure(config), start(), and stop(). Plugins receive a frozen, immutable PluginSDK object.
  3. MCP servers are connected via registerMcpTools(), translating MCP tool schemas into registry entries. Both stdio and HTTP MCP transports are supported.

After loading, a PluginWatcher monitors external plugin directories for file changes and performs hot-reload via replacePluginTools(), which atomically swaps old tool definitions for new ones without restarting.

Plugins communicate with the host application through a hook registry. Available hooks include message:receive, before:tool_call, after:tool_call, session:start, session:end, and others. Modifying hooks can mutate events; observing hooks are read-only.

Memory System

All persistent state lives in a single SQLite database at ~/.teleton/memory.db, managed by the MemorySystem module (src/memory/index.ts). See the Memory System page for the full schema and retrieval architecture.

The database is opened once at startup via getDatabase() (singleton) and shared across all subsystems. When an embedding provider is configured, sqlite-vec virtual tables (knowledge_vec, tg_messages_vec) enable cosine-similarity vector search alongside FTS5 full-text search.

WebUI and API Server

The optional WebUI is a Hono HTTP server (src/webui/server.ts) serving a React SPA. It exposes endpoints for inspecting sessions, memory, tool configurations, plugin management, and live chat. The API server (src/api/server.ts) provides a REST interface consumed by the WebUI and usable independently.

Both servers share the same TeletonApp instance via a dependency object built by buildServerDeps(), giving them read-write access to the agent runtime, tool registry, bridge, and memory system.

Startup Sequence

TeletonApp.start() initializes all subsystems in the following order:

  1. loadConfig() - Load and validate config.yaml
  2. new ToolRegistry() - Create registry in configured mode
  3. registerAllTools() - Register all built-in tools
  4. new AgentRuntime() - Create runtime with tool registry
  5. createBridge() - Instantiate user or bot bridge
  6. initializeMemory() - Open SQLite, run migrations, initialize embedder
  7. loadModules() - Load built-in capability modules
  8. PluginOrchestrator.loadAll() - Load external plugins and MCP servers
  9. registry.loadConfigFromDB() - Apply runtime tool config overrides
  10. Tool RAG index init - Index all tools for semantic search
  11. bridge.connect() - Authenticate with Telegram
  12. bridge.getMe() - Resolve own user ID
  13. bridge.onNewMessage() - Register message handler
  14. HeartbeatRunner.start() - Start periodic heartbeat timer
  15. ScheduledTaskHandler.start() - Start scheduled task poller
  16. WebUI / API server start - Bind HTTP listeners (if enabled)

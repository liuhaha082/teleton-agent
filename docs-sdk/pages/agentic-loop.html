Agentic Loop

A complete trace of how a Telegram message travels from the MTProto/Bot API event through the handler pipeline into the LLM tool loop and back out as a Telegram reply.

Pipeline Overview

A message travels through the following stages in order:

  1. Telegram event received via MTProto or Bot API
  2. ITelegramBridge.onNewMessage() delivers the event to the handler
  3. MessageHandler.handleMessage() runs pre-processing:
    1. Dedup check (in-memory Set plus offset store)
    2. Store to SQLite feed (tg_messages, tg_chats, tg_users)
    3. Fire plugin onMessage hooks (fire-and-forget)
    4. analyzeMessage() policy gate
    5. Rate limiter check
    6. ChatQueue.enqueue() to serialize per chat
  4. Inside the queue (typing simulation starts):
    1. Collect pending group history
    2. Resolve reply context
    3. Auto-transcribe voice/audio
  5. AgentRuntime.processMessage() drives the LLM turn:
    1. User keyword hooks (block or inject context)
    2. Plugin message:receive hook (block, mutate text, or inject context)
    3. Session load or create
    4. Session reset check (daily policy)
    5. Compute query embedding (concurrent)
    6. ContextBuilder.buildContext() - hybrid RAG (concurrent)
    7. Plugin prompt:before hook (concurrent)
    8. buildSystemPrompt() - assemble full system prompt
    9. Tool selection (Tool RAG or full list)
    10. LLM call and tool loop
    11. Append to transcript, update session
  6. Response dispatch (MessageHandler):
    1. Silent reply (__SILENT__): suppress, nothing sent
    2. Streamed response: already sent token by token, no further action
    3. Telegram send tool called: store to feed only
    4. Plain text, no send tool: call bridge.sendMessage() and store to feed

Phase 1: Message Handler

Deduplication

The handler maintains a bounded in-memory Set<string> of "chatId:messageId" keys (max 500 entries, oldest half evicted when full). GramJS can fire the same NewMessage event across multiple MTProto update channels; this set prevents double-processing.

In user mode, a secondary check reads the persistent offset store: if the incoming message ID is less than or equal to the stored offset for that chat, the message is skipped. The offset is written only after successful handling, so a crash before completion will reprocess the message on restart.

In bot mode, Grammy deduplicates via update_id internally; offset tracking is skipped.

analyzeMessage()

This method decides whether the agent should respond. It checks, in order:

  1. Message ID versus the stored offset (user mode only).
  2. Whether the sender is a bot (message.isBot): bots are never processed.
  3. For DMs: dm_policy in config (disabled, admin-only, allowlist, or open).
  4. For groups: group_policy (disabled, admin-only, allowlist, or open), then the require_mention flag.

If shouldRespond is false, the message is still stored to the feed. For groups where the reason is "not mentioned", the text is buffered in PendingHistory so it can be included as context when the next mention arrives.

Rate Limiting

A RateLimiter instance enforces two independent limits:

  • Global: rate_limit_messages_per_second, a sliding 1-second window across all chats.
  • Per-group: rate_limit_groups_per_minute, a sliding 60-second window per group chat ID. Stale entries are evicted when the map exceeds 100 groups.

ChatQueue

ChatQueue ensures messages in the same chat are processed serially. Internally it maintains a Map<chatId, Promise<void>> chain: each new message for a chat is chained onto the previous promise. Up to 10 chats may process simultaneously; additional chats wait via a FIFO wait queue (backpressure).

On graceful shutdown, chatQueue.drain() waits for all active chains to settle before the process exits.

Phase 2: Pre-processing (inside the queue)

Pending Group Context

When the agent is mentioned in a group, PendingHistory.getAndClearPending(chatId) returns any messages that arrived since the last response and were buffered because the agent was not mentioned. These messages are prepended to the formatted user message so the LLM sees the full conversation context.

Reply Context

If the incoming message is a reply, bridge.fetchReplyContext() resolves the original message text and sender name from the raw MTProto object. This context is injected into the message envelope so the LLM knows what is being replied to.

Voice Transcription

Voice and audio message types trigger an automatic call to telegramTranscribeAudioExecutor before the message reaches the LLM. On success, the transcribed text is prepended to the effective message body with a 🎤 (voice): prefix. Transcription failures are logged as warnings and do not block the response.

Phase 3: Agent Runtime (processMessage)

A: User Hook Evaluation

UserHookEvaluator reads a keyword blocklist and context-injection rules from the user_hook_config SQLite table. If the message text matches a blocked keyword, processMessage returns immediately with an empty response. Otherwise, any matching context rules append additional text to the system prompt.

B: Plugin message:receive Hook

If a hook runner is active, the message:receive event is dispatched to all plugins that registered for it. This is a modifying hook: plugins can set event.block = true to suppress the response entirely, mutate event.text to rewrite the message, or populate event.additionalContext to inject text into the system prompt.

C/D: Session Management

Sessions are keyed by chatId and stored in the sessions SQLite table. The runtime calls getOrCreateSession(chatId) to retrieve the session record, then checks the configured session_reset_policy (e.g., daily reset). On a reset, the current transcript is archived and an LLM-assisted summary is saved to memory before a new session starts.

E/F: Concurrent Embedding and Context Retrieval

Two operations run concurrently to minimise latency:

  • Embedding: the user message (concatenated with the last 3 user messages for richer context) is embedded using the configured provider. The embedding is used for both Tool RAG and feed search.
  • ContextBuilder.buildContext(): runs a hybrid search (FTS5 + vector if enabled) over the knowledge table (agent memory and knowledge files) and the tg_messages feed. Returns the top relevant chunks, capped by CONTEXT_MAX_RECENT_MESSAGES and CONTEXT_MAX_RELEVANT_CHUNKS. Trivial messages (greetings, single words) skip this step.

H: System Prompt Assembly

buildSystemPrompt() assembles the prompt from several layers, ordered for Anthropic prefix cache stability (most stable first, most dynamic last):

  1. SOUL.md - personality, rules, response format (most stable)
  2. SECURITY.md - owner-defined security rules
  3. STRATEGY.md - trading and business rules
  4. Owner identification block
  5. IDENTITY.md - agent self-description
  6. USER.md - user profile
  7. Bot mode capabilities block (if mode is bot)
  8. Memory context - frozen snapshot of core blocks, MEMORY.md, and daily logs
  9. RAG context - retrieved knowledge and feed history
  10. Current user identification
  11. Memory flush warning (if context is near limit)
  12. Heartbeat checklist (heartbeat turns only)
  13. Safety reminder (<reminder> tag, recency bias) - most dynamic

The memory context is frozen into a snapshot at the start of each session (captureMemorySnapshot()) and reused on every turn, so mid-session memory_write calls do not invalidate the Anthropic prompt cache prefix.

I: Tool Selection

If Tool RAG is enabled (tool_rag.enabled = true in config), registry.getForContextWithRAG() is called. It first applies scope and mode filters, then uses the query embedding to find semantically relevant tools via the tool_index FTS and vector tables. Always-included tools (defined in tool_rag.always_include) are added unconditionally. If the search returns no results, the full scope-filtered list is used as fallback.

Without Tool RAG, registry.getForContext() returns all scope-appropriate tools directly.

J: LLM Call and Tool Loop

The tool loop runs until the model stops requesting tools:

  1. Call the LLM with the current messages, tool list, and system prompt.
  2. If the response stop_reason is end_turn, exit the loop.
  3. For each tool call in the response (processed concurrently, up to TOOL_CONCURRENCY_LIMIT):
    1. Fire the plugin before:tool_call hook (can mutate parameters).
    2. Run registry.execute(toolCall, context): scope check (defense-in-depth), JSON schema validation, executor wrapped in a Promise.race against the timeout, result truncated to MAX_TOOL_RESULT_SIZE.
    3. Fire the plugin after:tool_call hook.
  4. Append tool results to the messages array and continue the loop.

If the token count exceeds the compaction threshold, CompactionManager either truncates old messages or requests an LLM-generated summary before the next iteration.

Tool results larger than MAX_TOOL_RESULT_SIZE bytes are truncated with a notice to the LLM. Old tool results in long conversations are progressively masked (replaced with a size summary) by maskOldToolResults() to reduce context pressure while preserving assistant reasoning.

Streaming

In bot mode, if config.telegram.stream_mode is "all" or "replace", streamWithContext() is used instead of chatWithContext(). The bridge's streamResponse() method sends tokens to Telegram incrementally via message draft updates, giving the user a typing-like experience for long responses.

Phase 4: Response Dispatch

After processMessage() returns, MessageHandler decides how to deliver the response:

ConditionAction
Response text is __SILENT__Suppress: nothing is sent to Telegram
response.streamed == trueAlready sent token by token; no further action
A Telegram send tool was called (e.g., telegram_send_message)Tool already sent the message; store the text to feed for history continuity
Non-empty plain text, no send toolCall bridge.sendMessage() with a reply to the original message; store response to feed

After dispatch, the offset is written for user mode, the per-chat typing interval is cleared, and the pending group history is cleared. Any unhandled exception during the queue task is caught and logged without crashing the process.

Heartbeat Turns

The HeartbeatRunner fires AgentRuntime.processMessage() on a configurable timer without a user message. The system prompt includes the HEARTBEAT.md checklist and an instruction to work through it step by step. The agent may make tool calls, write to memory, and send messages autonomously. If no action is needed, it replies NO_ACTION, which the heartbeat runner recognises and discards without sending anything to Telegram.

Heartbeat turns use a separate virtual chat ID (heartbeat) so they do not interfere with real conversation sessions.

Memory System

Teleton Agent stores all persistent state in a single SQLite database at ~/.teleton/memory.db (schema v1.18.0). The database combines FTS5 full-text search, sqlite-vec cosine-similarity vector tables, and a structured knowledge layer to implement a hybrid RAG retrieval pipeline.

Four Memory Layers

The agent is aware of four distinct memory layers, each with different lifetime and capacity characteristics:

LayerWrite targetCapacityLifetimeLoaded in prompt
Core Memorymemory_write target=core~3 000 chars total across structured blocks (identity, preferences, lessons, goals, contacts)PermanentYes, every session
Persistent Memory (MEMORY.md)memory_write target=persistentSoft limit 80 lines, hard truncation at 150 linesPermanentYes, every session (fallback if no core blocks)
Daily Logsmemory_write target=dailyUnbounded on diskSession notes; yesterday + today loadedYes, yesterday and today
Session Search (feed)Automatic (all messages stored)UnboundedPermanentNo, retrieved on demand via session_search tool or RAG

The agent is instructed to write to memory only when it learns something that changes future behaviour: new contacts, lessons from mistakes, user preferences. Price snapshots, task progress, and heartbeat logs are not stored persistently.

Memory in the System Prompt

loadMemoryContext() in src/soul/loader.ts assembles the memory block injected into every system prompt:

  1. Calls getCoreMemoryForPrompt() to retrieve structured core blocks (identity, preferences, lessons, goals, contacts). If any blocks exist, they form the ## Core Memory section.
  2. If no core blocks exist, falls back to raw MEMORY.md content (max 150 lines, truncated with a warning if exceeded).
  3. Appends yesterday's and today's daily log entries as a separate section.

This context is frozen into a snapshot at the start of each session via captureMemorySnapshot(). Mid-session memory_write calls update the disk file but the snapshot remains unchanged for the session's lifetime, preserving the Anthropic prefix cache across all turns. The snapshot is cleared when a session resets, so the next session picks up any writes.

Database Schema

The schema is defined in src/memory/schema.ts and applied via ensureSchema() at startup. Migrations are run forward-only in version order up to the current version.

knowledge

The central table for agent memory. Each row is a text chunk with an optional embedding vector.

knowledge table
CREATE TABLE knowledge (
  id          TEXT PRIMARY KEY,
  source      TEXT NOT NULL CHECK(source IN ('memory', 'session', 'learned')),
  path        TEXT,            -- file path for memory/session sources
  text        TEXT NOT NULL,   -- chunk content
  embedding   TEXT,            -- stored embedding (legacy JSON; vec stored in knowledge_vec)
  start_line  INTEGER,
  end_line    INTEGER,
  hash        TEXT NOT NULL,   -- SHA of text, used for change detection
  created_at  INTEGER NOT NULL DEFAULT (unixepoch()),
  updated_at  INTEGER NOT NULL DEFAULT (unixepoch()),
  -- added in v1.17.0:
  importance  REAL DEFAULT 0.5,
  access_count INTEGER DEFAULT 0,
  last_accessed_at INTEGER,
  status      TEXT DEFAULT 'active',
  memory_type TEXT DEFAULT 'semantic'
);

FTS5 full-text search is provided by the knowledge_fts virtual table, kept in sync via INSERT/DELETE/UPDATE triggers. When vector search is enabled, embedding vectors are stored separately in the knowledge_vec vec0 virtual table (cosine distance metric, dimension count matches the configured embedding model).

tg_messages, tg_chats, tg_users

The Telegram feed. Every message seen by the agent is stored here regardless of whether it responds. This feed is the source for the session_search tool and for feed history retrieval in ContextBuilder.

tg_messages table (key columns)
CREATE TABLE tg_messages (
  id           TEXT PRIMARY KEY,
  chat_id      TEXT NOT NULL REFERENCES tg_chats(id) ON DELETE CASCADE,
  sender_id    TEXT REFERENCES tg_users(id) ON DELETE SET NULL,
  text         TEXT,
  embedding    TEXT,           -- legacy; vectors in tg_messages_vec
  reply_to_id  TEXT,
  is_from_agent INTEGER DEFAULT 0,
  has_media    INTEGER DEFAULT 0,
  media_type   TEXT,
  timestamp    INTEGER NOT NULL
);

tg_messages_fts provides full-text search. FTS triggers skip rows where text IS NULL (media-only messages). tg_messages_vec provides cosine-similarity vector search on message embeddings when an embedding provider is configured.

sessions

One row per conversation session (keyed by chat_id). Tracks the session UUID, message count, token usage, the LLM model and provider in use, daily reset date, and a compressed summary generated on session expiry.

Conversation transcripts are stored as separate files (not in SQLite) under ~/.teleton/sessions/.

tool_index

Used by the Tool RAG system. One row per registered tool, containing the tool's name, description, and a concatenated search_text field. The tool_index_fts virtual table provides keyword search over tool descriptions. When vector search is enabled, tool embeddings are stored in a tool_index_vec table.

tool_index table
CREATE TABLE tool_index (
  name        TEXT PRIMARY KEY,
  description TEXT NOT NULL,
  search_text TEXT NOT NULL,   -- concatenation of name, description, parameters
  updated_at  INTEGER NOT NULL DEFAULT (unixepoch())
);

Embedding Providers

The embedding system is configured via config.embedding. Three providers are supported:

ProviderHow it runsDefault modelDimensions
localIn-process via @xenova/transformers (ONNX)Xenova/all-MiniLM-L6-v2384
anthropicAnthropic Embeddings APIvoyage-3-lite512
noneNo-op (vector tables disabled)N/AN/A

All providers are wrapped in CachedEmbeddingProvider (unless provider is none), which stores computed vectors in the embedding_cache SQLite table keyed by (SHA256(text), model, provider). Cache hits are served from SQLite; misses call the underlying provider and are written back atomically.

When the embedding dimension changes (e.g., switching providers), the ensureVectorTables() function detects the mismatch, drops the old knowledge_vec and tg_messages_vec tables, and recreates them with the new dimension. Existing embeddings in embedding_cache for the old model remain but are no longer referenced.

Knowledge Indexer

KnowledgeIndexer (src/memory/agent/knowledge.ts) ingests workspace files into the knowledge table. At startup it scans:

  • ~/.teleton/workspace/MEMORY.md: the agent's persistent fact store (source: memory)
  • ~/.teleton/workspace/memory/*.md: daily log files (source: session)
  • Any facts the agent has written via the memory_write tool (source: learned)

Files are split into overlapping chunks. Each chunk is hashed; existing rows with the same hash are skipped. Changed or new chunks are embedded and inserted. Deleted lines cause the corresponding rows to be removed. This incremental approach avoids re-embedding the entire knowledge base on every restart.

Context Builder (Hybrid RAG)

ContextBuilder (src/memory/search/context.ts) is called once per incoming message turn to retrieve the most relevant context to inject into the system prompt. It runs two searches in parallel:

Knowledge search

Searches the knowledge table for agent memory and indexed workspace files. When vector search is available and a query embedding exists, it uses cosine similarity on knowledge_vec. It falls back to FTS5 keyword search on knowledge_fts otherwise. Results are ranked and capped at CONTEXT_MAX_RELEVANT_CHUNKS.

Feed history search

Retrieves the most recent messages from tg_messages for the current chat (capped at CONTEXT_MAX_RECENT_MESSAGES). For DMs, a broader cross-chat semantic search is also run to surface relevant context from other conversations.

Result injection

Retrieved chunks are formatted and appended to the ## Context section of the system prompt. This is the dynamic, non-cached portion of the prompt. It changes every turn and intentionally sits outside the prefix cache window.

Tool RAG

With large numbers of tools, sending the full tool list to the LLM on every turn is expensive and may exceed provider limits. Tool RAG solves this by indexing all tool descriptions and selecting only the tools most relevant to the current message.

ToolIndex (src/agent/tools/tool-index.ts) maintains the tool_index and tool_index_fts tables. The search path in registry.getForContextWithRAG():

  1. Apply scope and mode filters to get the candidate set.
  2. Add all tools marked as always-included (configured in tool_rag.always_include).
  3. Run FTS keyword search + optional vector similarity search on the query.
  4. Return the union of always-included and search results, capped at tool_rag.top_k.
  5. If search returns no results, fall back to the full scope-filtered list.

The tool index is updated automatically when plugins are loaded or hot-reloaded via the registry.onToolsChanged() callback.

Providers with no tool limit (e.g., some OpenAI-compatible endpoints) can optionally skip Tool RAG entirely via tool_rag.skip_unlimited_providers, sending the full tool list directly.

Context Compaction

Long conversations accumulate tokens. CompactionManager monitors context size after each LLM turn. It operates in two stages:

  1. Soft threshold: when context tokens exceed a configurable fraction of the model's context window, a memoryFlushWarning is added to the system prompt, prompting the agent to write important facts to memory before the window overflows.
  2. Hard compaction: when the message count or token count exceeds hard limits, the runtime either truncates old messages from the context or requests an LLM-generated summary of the conversation so far, then replaces the old messages with the summary. The session transcript is archived and a new one starts.

Old tool results in long contexts are progressively masked by maskOldToolResults(), replacing large result payloads with a brief size notice to reduce context pressure without losing the record that the tool was called.

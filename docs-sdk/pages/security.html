Security Model

Security in Teleton Agent is enforced at multiple independent layers: message admission, tool scope filtering, execution-time checks, execution sandboxing, plugin isolation, and prompt-level safety rules. No single layer is assumed to be complete.

Access Control (Message Admission)

Before any message reaches the agent, MessageHandler.analyzeMessage() applies the configured access policy. All policies are set in config.yaml under the telegram: key.

Admin IDs

telegram.admin_ids is a list of Telegram user IDs that receive elevated trust. Admins bypass dm_policy and group_policy restrictions and can always execute admin-only and allowlist-scoped tools. This list is matched against message.senderId (numeric ID, not username, as usernames can change).

DM Policy

telegram.dm_policy controls who can initiate private conversations with the agent:

ValueBehaviour
disabledNo DMs accepted from anyone (including admins)
admin-onlyOnly users in admin_ids
allowlistUsers in admin_ids or allow_from
openAll Telegram users

Group Policy

telegram.group_policy controls which groups the agent participates in:

ValueBehaviour
disabledAgent ignores all group messages
admin-onlyOnly responds in groups to admin-originated messages
allowlistOnly responds in groups listed in group_allow_from
openAll groups (subject to require_mention)

telegram.require_mention (default: true) forces the agent to only respond in groups when explicitly mentioned, preventing accidental responses to every message in active groups.

Tool Scope Enforcement

Tool scopes are enforced at two independent points: once in registry.getForContext() when building the tool list sent to the LLM, and again in registry.execute() when the LLM actually calls the tool. The second check is defense-in-depth against jailbroken or manipulated model outputs.

ScopeLLM sees toolExecution allowed
always / openAll contextsAll contexts
dm-onlyDMs onlyDMs only
group-onlyGroups onlyGroups only
admin-onlyAdmin senders onlyAdmin senders only
allowlistAdmins + allow_from listAdmins + allow_from list
disabledNeverNever (returns error)

Scopes can be overridden per-tool at runtime via the tool_config SQLite table. The WebUI and admin commands expose this without requiring a config file edit or restart. Tools can also be individually disabled (hidden from the LLM entirely) without changing their scope.

Module-Level Group Permissions

ModulePermissions provides an additional layer for group chats. Each tool belongs to a module (derived from the tool name prefix). Per-group, each module can be set to open, admin (only admins can trigger it), or disabled. This is stored in the database and checked at both the visibility and execution stages.

Mode Restrictions

Some tools only make sense in one bridge mode. Tools registered with requiredMode: "user" (e.g., browsing dialogs, reading full history) are filtered from the tool list and blocked at execution when the bridge is in bot mode. This prevents the LLM from hallucinating successful user-mode operations when running as a bot.

Tool Execution Safety

Parameter Validation

Before any tool executor runs, validateToolCall() from @mariozechner/pi-ai validates the LLM-supplied arguments against the tool's JSON Schema. Invalid or missing required parameters cause the call to fail before any side effect occurs.

Execution Timeout

Every tool executor is wrapped in a Promise.race() against a timeout (TOOL_EXECUTION_TIMEOUT_MS). A timed-out tool call returns a structured error to the LLM; it does not crash the runtime or leave the conversation in a broken state.

Result Truncation

Tool results larger than MAX_TOOL_RESULT_SIZE bytes are truncated before being appended to the conversation context. A note is included in the truncated result so the LLM knows the output was cut. This prevents excessively large outputs from consuming the entire context window.

Exec Capability

The exec module (src/agent/tools/exec/) allows the agent to run arbitrary shell commands on the host. It is the most powerful and potentially most dangerous capability in the system. Several controls apply.

Linux-Only

The exec module checks process.platform !== "linux" at load time and disables all exec tools on non-Linux systems. This is not a security boundary but a deliberate capability restriction: exec is designed for Linux server deployments only.

Scope Restriction

config.capabilities.exec.scope determines who can trigger exec tools:

ValueBehaviour
admin-onlyOnly users in admin_ids (default)
allowlistUsers in admin_ids or the exec-specific allow_from list
allAll users (not recommended)

Setting exec.mode = "off" in config disables all exec tools completely and prevents them from appearing in the tool registry.

Configurable Limits

config.capabilities.exec.limits caps the blast radius of any single command:

  • timeout: maximum wall-clock seconds per command (1–3600). When exceeded, the child process receives SIGTERM, then SIGKILL after a 5-second grace period.
  • max_output: maximum bytes of combined stdout/stderr captured. Output beyond this is truncated and flagged in the result.

Processes are spawned with detached: true and killed via process.kill(-pid, signal) (negating the PID targets the entire process group), so subprocesses spawned by the command are also terminated on timeout.

Audit Trail

When exec.audit.log_commands = true (default), every exec tool call is recorded to the exec_audit SQLite table before execution begins. The row captures the requesting user ID, username, command text, and timestamps. On completion, exit code, stdout, stderr, duration, and truncation status are written back. This log is immutable (append-only) and is visible in the WebUI.

exec_audit table
CREATE TABLE exec_audit (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  timestamp   INTEGER NOT NULL DEFAULT (unixepoch()),
  user_id     INTEGER NOT NULL,
  username    TEXT,
  tool        TEXT NOT NULL,      -- exec_run | exec_install | exec_service | exec_status
  command     TEXT NOT NULL,
  status      TEXT NOT NULL DEFAULT 'running'
    CHECK(status IN ('running','success','failed','timeout','killed')),
  exit_code   INTEGER,
  signal      TEXT,
  duration_ms INTEGER,
  stdout      TEXT,
  stderr      TEXT,
  truncated   INTEGER NOT NULL DEFAULT 0
);

Plugin Isolation

External plugins run in the same Node.js process but are isolated from core internals and from each other through several mechanisms:

Frozen SDK Object

Each plugin receives a PluginSDK object constructed specifically for that plugin and then frozen via Object.freeze() recursively. Plugins cannot reassign SDK properties, replace methods, or add new ones. A plugin cannot reach into another plugin's SDK instance.

Isolated SQLite Database

Plugins that export a migrate(db) function receive their own SQLite database file (separate from the main memory.db). Plugins that use sdk.storage receive a key-value namespace scoped to their plugin name. No plugin has direct access to the main database tables.

Secrets via Environment Variables

Plugin secrets are declared in the PluginManifest.secrets field and resolved from environment variables at load time. The naming convention is ${PLUGIN_NAME_UPPER}_${KEY_UPPER}. The raw environment is not passed to plugins; only explicitly declared secrets are surfaced through sdk.secrets.

Sanitized Config

Plugins receive a sanitized view of the application config via sdk.config, not the raw config object. Sensitive fields (API keys, admin IDs, wallet mnemonics) are not included in the plugin-facing config.

Tool Name Collision Prevention

The registry tracks which tool names belong to which plugin via pluginToolNames. During hot-reload, replacePluginTools() prevents a plugin from overwriting tools registered by a different plugin or by the core. An attempt to do so is logged as a warning and the conflicting tool is skipped.

Prompt Injection Defenses

User-supplied text flows through several sanitization steps before reaching the LLM system prompt or conversation context.

Input Sanitization

Two utility functions are applied to all untrusted text before injection into prompts:

  • sanitizeForPrompt(text): removes or escapes characters and sequences that could break out of string boundaries in the system prompt.
  • sanitizeForContext(text): applied to memory content, user profiles, and retrieved context chunks before they are embedded in the prompt. Prevents a malicious entry in the agent's own memory files from injecting instructions.

These functions are applied to: sender names, usernames, message text after the message:receive hook, hook-injected context, identity/user/memory file contents, and retrieved RAG chunks.

Keyword Blocklist (UserHookEvaluator)

UserHookEvaluator reads blocklist rules from the user_hook_config SQLite table. Rules are evaluated against the raw message text before any LLM processing. A matched blocklist entry returns an empty response immediately; the message never reaches the agent runtime. Rules can be added, modified, or removed at runtime without restarting.

Plugin message:receive Hook

Plugins can participate in input validation via the message:receive hook. A plugin can set event.block = true with a reason, preventing the message from being processed. This is used by moderation plugins to implement custom filtering logic without modifying the core.

Wallet and Fund Protection

The agent's default soul (SOUL.md) contains hard-coded safety rules that the LLM is instructed to follow:

  • Reversible actions (reading data, checking balances, searching) can be performed freely.
  • Irreversible actions (sending messages to third parties, transferring TON or jettons, swapping tokens, sending gifts, making purchases) require explicit owner confirmation before execution. The agent must state the exact amount, destination, and action, and wait for approval.
  • The agent will not send funds even if a message claims to be from the creator or owner. Identity cannot be verified through a chat message.
  • Self-preservation and unsolicited resource acquisition are explicitly prohibited.

These rules are the last section of the system prompt (before the final <reminder> tag), taking advantage of recency bias in LLM attention. The <reminder> tag containing the fund-transfer confirmation rule is always the final line of every system prompt.

In bot mode, the soul instructs the agent to use telegram_send_buttons with Confirm/Cancel inline buttons for any transaction, creating an additional confirmation step before the transfer tool is called.

Message Pipeline Safety

Deduplication

Two independent dedup layers prevent the same message from being processed twice:

  1. An in-memory Set of up to 500 recent "chatId:messageId" strings. Catches duplicate events fired within the same session.
  2. A persistent per-chat offset store. The highest processed message ID is written to disk after each successful response. On reconnect or restart, messages with IDs at or below the stored offset are skipped.

The offset is written after successful handling (not before), so a crash mid-response will reprocess the message rather than silently drop it.

Rate Limiting

A two-tier RateLimiter prevents flooding:

  • Global: enforces a maximum number of outgoing messages per second across all chats.
  • Per-group: enforces a maximum number of responses per group per minute. Stale group entries are evicted to prevent memory growth in deployments with many groups.

Per-Chat Serialization

ChatQueue serializes message processing per chat via promise chaining. This prevents race conditions where two simultaneously arriving messages in the same chat both start processing, read the same session state, and then overwrite each other's context writes. At most 10 chats process concurrently; additional chats queue behind a semaphore.

Graceful Shutdown

On SIGINT or SIGTERM, TeletonApp.stop() calls chatQueue.drain(), waiting up to SHUTDOWN_TIMEOUT_MS for all in-flight message processing to complete before closing the database and disconnecting the bridge. This prevents partial writes to the session transcript or memory database on shutdown.
